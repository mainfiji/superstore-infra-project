## 📝 Как я выполнял проект (подробно)

Проект разбит на три логические части. Ниже я описываю **каждый шаг** — какие команды вводил, какие конфигурации применял и как проверял результат. Все проверки подтверждены скриншотами.

### Часть 1. Сеть (13 заданий)

**Задание 1. VLAN на коммутаторах даркстора**  
Я настроил VLAN 30 (Logistics), 40 (Storage) и 60 (DS-Servers) на трёх коммутаторах. Порты, к которым подключены пользовательские устройства, перевёл в режим access с соответствующим VLAN. Между коммутаторами и маршрутизатором R-Darkstore настроил trunk-порты с разрешёнными VLAN 30,40,60. Затем вручную назначил клиентам статические IP (например, PCL-1 получил 10.10.3.1/24) и проверил связность ping внутри одного VLAN — все пинги прошли успешно.

**Задание 2. Маршрутизация на R-Darkstore**  
На маршрутизаторе создал субинтерфейсы:  
- Ethernet0/0.30 с IP 10.10.3.1/24 (шлюз для VLAN 30)  
- Ethernet0/0.40 с IP 10.10.4.1/24 (шлюз для VLAN 40)  
- Ethernet0/1.60 с IP 10.10.6.1/24 (шлюз для VLAN 60)  

Проверил, что клиенты пингуют свои шлюзы, а также устройства из разных VLAN (например, PCL-3 → PCS-3) — маршрутизация работает.

**Задание 3. DHCP на R-Darkstore**  
Настроил DHCP-пулы для VLAN 30 и 40 на том же маршрутизаторе. Удалил статические адреса на клиентах и перезапросил IP через `ip dhcp` — устройства получили адреса из пулов автоматически.

**Задание 4. Выход в интернет (даркстор)**  
Настроил интерфейс Ethernet0/2 R-Darkstore с IP 55.55.55.105/30, а на ISP — 55.55.55.106/30. Добавил маршрут по умолчанию `ip route 0.0.0.0 0.0.0.0 55.55.55.106`. Пинги между R-Darkstore и ISP успешны.

**Задание 5. Тестовый интернет-сервер**  
На ISP создал дополнительный интерфейс Ethernet0/2 с IP 1.1.1.2/24 и подключил к нему VPC с адресом 1.1.1.1. Пинги с ISP до 1.1.1.1 проходят.

**Задание 6. NAT в дарксторе**  
Настроил PAT (overload) с помощью ACL 11, разрешающего все внутренние подсети, и применил его к внешнему интерфейсу. Проверил, что PCL-1 и PCS-3 пингуют 1.1.1.1 — NAT работает. Статистика `show ip nat statistics` показывает хиты.

**Задание 7. VLAN в головном офисе**  
Аналогично даркстору, настроил VLAN 10 (IT), 20 (Management), 50 (HQ-Servers) на SW1-HQ и SW2-HQ, trunk-порты и статические IP для проверки. Пинги между PCI-1 и PCI-2, FS и Mail, PCM-1 и PCM-2 успешны.

**Задание 8. Маршрутизация на R-HQ**  
Создал субинтерфейсы:  
- Ethernet0/2.10 с IP 10.10.1.1/24  
- Ethernet0/2.20 с IP 10.10.2.1/24  
- Ethernet0/1.50 с IP 10.10.5.1/24  

Проверил пинги между разными VLAN (PCI-1 → PCM-1, PCM-2 → DC, PCI-2 → DNS) — всё работает.

**Задание 9. DHCP в головном офисе**  
Настроил DHCP-пулы для VLAN 10 и 20 на R-HQ. Клиенты получили адреса автоматически (PCI-1 и PCM-1).

**Задание 10. Выход в интернет (головной офис)**  
Настроил стык R-HQ (55.55.55.101/30) и ISP (55.55.55.102/30), добавил маршрут по умолчанию. Пинги и таблица маршрутизации подтверждают связность.

**Задание 11. NAT в головном офисе**  
PAT через ACL 11 на интерфейсе Ethernet0/0. Проверил, что PCI-1 и PCM-1 пингуют 1.1.1.1. Статистика NAT показывает успешные трансляции.

**Задание 12. Публикация веб-сервера**  
Настроил статический DNAT: `ip nat inside source static tcp 10.10.5.60 80 55.55.55.101 8080 extendable`. Внешний порт 8080 проброшен на внутренний веб-сервер. Команда `show ip nat translations` показывает запись. (Telnet с ISP не ответил, но конфигурация корректна — скорее всего, веб-сервер не был запущен.)

**Задание 13. Связность между локациями**  
На WAN-маршрутизаторе настроил статические маршруты:  
- к сетям HQ (10.10.1.0/24, 10.10.2.0/24, 10.10.5.0/24) через 192.168.10.1  
- к сетям Darkstore (10.10.3.0/24, 10.10.4.0/24, 10.10.6.0/24) через 192.168.20.1  

На R-HQ и R-Darkstore добавил соответствующие маршруты до удаленных подсетей. Пинги между PCI-1 (HQ) и PCS-3 (Darkstore), PCM-1 и PCL-1, DC и FS проходят — сквозная маршрутизация работает.
